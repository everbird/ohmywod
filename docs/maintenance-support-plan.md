# 站点维护成本支持计划（草案）

> 本文是战报网收入与用户支持方向的工作计划，不是营收增长方案。目标是为个人兴趣站点提供一种自愿、低维护、不过度打扰用户的维护成本补贴方式。
>
> **当前方向：停止扩展广告位，推荐撤掉全部 AdSense，把微信打赏和爱发电作为主要支持路径。核心功能继续免费，不设置付费墙，也不把支持金额变成继续运营的硬承诺。**

## 1. 背景与结论

### 1.1 站点性质

- 这是面向中文 World of Dungeons 玩家、小众且以回访用户为主的个人兴趣工具站。
- 站点价值来自上传、整理、搜索、分享和阅读战报，不来自内容消费规模或广告流量。
- AdSense 只是实验性尝试。国内流量 coverage 长期只有个位百分比，站长已判断其月收入肯定不到月维护成本的 10%；本计划不再要求补采数据来证明是否应转向。
- 现有支持入口只有首页微信打赏二维码，缺少可持续支持渠道和独立、清楚的说明。

### 1.2 当前实现

- `ohmywod/templates/base.html` 在桌面侧边栏加载一个 AdSense 单元。
- `ohmywod/templates/report_reader.html` 已在战报内容尾部加载一个 AdSense 单元。它是旧广告方案撰写后实际落地的部分。
- `ohmywod/templates/landing.html` 同时引导微信打赏、参与 GitHub 和“点广告”。其中鼓励点击广告的文案有平台政策风险，也与新的支持方向冲突。
- `ohmywod/templates/base_noadsense.html` 通过模板覆盖区分无广告页面。全部移除 AdSense 后，这层差异可能不再需要。
- 旧广告优化方案已归档到 `docs/archive/ad-optimization-plan-2026-07.md`，只保留为历史记录。

### 1.3 已拍板的原则

1. 收入定位是“维护成本支持”，不是利润最大化，也不以覆盖全部成本作为成功前提。
2. 不再增加移动端锚定广告、插页广告、详情页横幅、列表 in-feed 或其他 Auto Ads 格式。
3. 推荐终态是完全移除 AdSense。若未来重新考虑，只允许单独评估阅读模式内容尾部这一种低干扰位置，不默认恢复其他广告。
4. 核心功能保持免费：上传、浏览、搜索、分享、收藏和阅读体验不因是否支持而分层。
5. 支持必须自愿，不使用倒计时、弹窗、遮挡、反复催促、羞辱式文案或“点广告支持本站”之类引导。
6. 回馈以低维护、非排他的内容为主，不承诺定制功能、优先修复、专属客服或影响站点路线的特权。
7. 渠道和文案宁少勿杂。每增加一种渠道，都要考虑手续费、提现、身份信息和长期维护负担。

### 1.4 成功判断

这项计划不设广告 RPM、coverage 或转化率目标。满足以下结果即可认为方向成立：

- 站内不再加载商业广告，核心页面也没有广告造成的布局、性能和注意力干扰。
- 用户能在一个清楚的入口找到微信打赏和爱发电，并理解支持是自愿补贴维护成本。
- 支持渠道本身不产生明显运营负担；即使收入很少，也不需要为了“提高转化”破坏站点体验。
- 每隔一段较长时间可以轻量复盘支持金额与实际维护成本，但复盘结果不自动触发加广告或收费。

## 2. 这份计划怎么维护

这是个人兴趣项目，不需要 owner、RACI、增长实验平台或复杂财务流程。后续工作通常由一个 AI 工具推进，再由另一个 AI 工具独立检查。涉及站外账号、收款信息、公开成本或运营承诺时，由用户做最后决定。

每个事项有两个可选角色：

- `Drive AI`：调查现状、提出选项、实施改动，并更新本文和 changelog。
- `Review AI`：独立检查文案、diff、用户体验和剩余风险，不只复述 Drive AI 的结论。

角色可以写工具名，例如 `Codex` 或 `Claude Code`；尚未分配时写 `unassigned`。

### 2.1 状态枚举

工作项的 `状态` 只能使用以下值：

- `todo`：方向已确认，尚未开始
- `assessing`：仍需用户选择或外部信息，尚不能直接实施
- `in_progress`：正在调查或实施
- `blocked`：存在明确阻塞；必须写明解除方式
- `done`：完成判断已经满足，并有可复核证据
- `cancelled`：明确决定不做；记录理由和重新考虑的触发条件

### 2.2 更新规则

1. 每个 `SUP-NNN` ID 永久不变、不可复用。新增事项使用 front matter 的 `next_item_id`，并同步递增。
2. 开始工作前先读本文和 `git status`，保留用户或其他工具的未提交改动。
3. `状态` 是事项主真值。状态变化、代码改动和 changelog 放在同一波变更里。
4. `done` 需要可复核证据；只申请了账号、只写了文案或只删除一处广告都不算整个事项完成。
5. `blocked` 要说明卡在哪里、怎样解除。`cancelled` 要说明为什么不做，以及何时值得重看。
6. 不在本文记录收款账号、二维码原图、身份信息、订单明细或支持者个人信息。
7. 涉及爱发电账号、收款设置或其他站外操作时，先由用户确认并执行；本文不构成外部账号操作授权。
8. 每波改动都在文末按时间正序追加一条 changelog。旧记录不重写，纠错时追加新记录。
9. 每次变更更新 front matter 的 `last_updated`。

### 2.3 固定工作项格式

新增事项使用同一套结构：

- 元信息：`状态`、`优先级`、`波次`、`Drive AI`、`Review AI`、`依赖`、`最后更新`、`结论置信度`
- 内容：`问题与影响`、`证据`、`方向与要点`、`完成判断`、`Review 关注`、`执行证据`

优先级定义：

- `P0`：当前存在明确的平台政策风险，或继续运行会直接违背已拍板方向
- `P1`：建立维护成本支持主路径所必需
- `P2`：透明度、复盘或可选回馈；不影响主路径成立

## 3. 建议推进顺序

### Wave 0：结束 AdSense 实验

范围：SUP-001、SUP-002。

方向：先清除鼓励广告点击的文案，再从代码和 AdSense 后台撤掉广告。这个波次完成后，站点不再把用户注意力和页面空间作为收入来源。

### Wave 1：建立简单的支持入口

范围：SUP-003、SUP-004。

方向：保留微信打赏，新增爱发电作为可持续支持渠道，再把两者收敛到一个清楚、不打扰核心操作的入口。

### Wave 2：让支持关系可长期维持

范围：SUP-005、SUP-006。

方向：只增加真正能建立信任、且长期做得下去的成本说明和回馈方式。宁可保持简单，也不制造新的内容债务或服务承诺。

## 4. 工作项

### SUP-001 — 清除鼓励广告点击的文案

- 状态：`todo`
- 优先级：`P0`
- 波次：Wave 0
- Drive AI：`unassigned`
- Review AI：`unassigned`
- 依赖：无
- 最后更新：2026-07-16
- 结论置信度：`confirmed`

问题与影响：`landing.html` 的支持文案鼓励用户“偶尔点点网站左下角的广告”。这既与自愿维护成本支持的定位冲突，也会给 AdSense 账号带来不必要的平台政策风险。

证据：`ohmywod/templates/landing.html` 的“反馈与支持”区域。

方向与要点：

- 立即删除所有鼓励查看、点击或互动广告的站内文案。
- 在 AdSense 完全撤下前，支持文案只提微信打赏、爱发电准备状态和参与 GitHub，不把广告描述为支持方式。
- 全仓搜索类似表述，避免帮助页、页脚或历史模板遗漏。

完成判断：用户可见页面不再引导广告点击，且新的支持文案不使用压力式表达。

Review 关注：同时检查中文文案、图片 alt、模板注释和可能由 JavaScript 注入的内容。

执行证据：尚无。

### SUP-002 — 完全移除 AdSense 及其遗留配置

- 状态：`todo`
- 优先级：`P0`
- 波次：Wave 0
- Drive AI：`unassigned`
- Review AI：`unassigned`
- 依赖：SUP-001
- 最后更新：2026-07-16
- 结论置信度：`recommended`

问题与影响：当前至少有侧边栏和阅读模式页尾部两个广告单元。即使没有成功填充，它们仍会引入第三方脚本、额外请求、布局处理和长期政策维护，而预期收入不足以构成继续实验的理由。

证据：`ohmywod/templates/base.html`、`ohmywod/templates/report_reader.html`、`ohmywod/templates/base_noadsense.html` 和 `ohmywod/templates/ads.txt`。

方向与要点：

- 删除两个 AdSense 单元、加载脚本、初始化脚本及仅为广告存在的布局修补。
- 检查 `sidebar_ad` / `base_noadsense.html` 的继承关系。若差异只为 AdSense 服务，收敛模板；WoD 推广图是否保留作为非商业内容单独判断。
- 全仓确认不再包含 publisher ID、slot ID、`adsbygoogle` 或 Auto Ads 代码。
- 在 AdSense 后台关闭 Auto Ads 和现有广告单元；后台操作由用户完成，代码移除不等待后台操作。
- 核实 `ads.txt` 的实际服务方式。确认不再需要后删除，避免继续暗示站点参与广告投放。

完成判断：公开页面不再请求 AdSense 资源，桌面和移动端均无广告占位；源码中无仍会生效的广告配置；后台已停止自动广告。

Review 关注：检查模板继承、移动端、阅读模式和缓存版本，避免只删可见元素却继续加载脚本。

执行证据：尚无。

### SUP-003 — 建立爱发电支持渠道

- 状态：`assessing`
- 优先级：`P1`
- 波次：Wave 1
- Drive AI：`unassigned`
- Review AI：`unassigned`
- 依赖：SUP-001
- 最后更新：2026-07-16
- 结论置信度：`recommended, external setup required`

问题与影响：微信二维码适合一次性支持，但不适合表达持续维护意愿，也缺少统一的项目说明。爱发电更贴近中文用户，但账号、实名、收款和平台条款属于站外事项。

证据：当前仓库只有 `ohmywod/static/img/wechat-reward-code.jpg` 和首页微信打赏入口，没有持续支持链接。

方向与要点：

- 用户先确认愿意使用爱发电，并完成账号、主页和收款设置。
- 主页只说明站点用途、维护内容和自愿支持性质，不承诺稳定更新频率或永久运营。
- 初始只设置一到两个简单档位；所有档位表达支持程度，不绑定核心功能或人工服务。
- 上线前按当时条款确认手续费、提现、身份展示、退款和隐私规则，不把易变的平台数字写死在站内文案。
- 将爱发电 URL 作为配置或单一模板入口管理，避免多处硬编码。

完成判断：爱发电主页和收款路径可用，站内能够链接到正确页面，且承诺范围清楚、可长期履行。

Review 关注：检查页面是否泄露不必要身份信息，档位是否悄悄形成持续内容或客服义务。

执行证据：尚无；等待用户完成或确认站外设置。

### SUP-004 — 收敛站内“支持维护”入口

- 状态：`todo`
- 优先级：`P1`
- 波次：Wave 1
- Drive AI：`unassigned`
- Review AI：`unassigned`
- 依赖：SUP-002、SUP-003
- 最后更新：2026-07-16
- 结论置信度：`recommended`

问题与影响：首页现有支持面板把微信、广告和 GitHub 混在一段文字里。撤掉广告后，需要让金钱支持、问题反馈和代码贡献各自清楚，同时避免把首页做成募款页面。

证据：`ohmywod/templates/landing.html` 的“反馈与支持”区域。

方向与要点：

- 首页保留一个克制的“支持维护”入口，和反馈、联系方式并列，不弹窗、不悬浮、不阻断操作。
- 独立支持页或轻量弹层展示微信二维码、爱发电链接和简短说明；具体形态实施时结合现有路由和移动端布局决定。
- 明确说明：支持用于服务器、存储、域名等维护开销；不支持也不影响任何功能。
- GitHub 贡献保留为非金钱参与方式，但与收款按钮在视觉和语义上区分。
- 外链使用清楚的站点名称和安全属性；二维码在移动端提供可点击或可保存的替代路径。

完成判断：用户能在不超过两次操作内找到两种支持渠道，未支持用户不会遇到额外打扰，移动端文字和二维码可正常使用。

Review 关注：检查小屏布局、外链行为、无障碍文本，以及文案是否让用户误以为支持可以换取功能优先级。

执行证据：尚无。

### SUP-005 — 决定是否公开维护成本说明

- 状态：`assessing`
- 优先级：`P2`
- 波次：Wave 2
- Drive AI：`unassigned`
- Review AI：`unassigned`
- 依赖：SUP-004
- 最后更新：2026-07-16
- 结论置信度：`optional`

问题与影响：适度说明服务器、存储和域名开销有助于用户理解支持用途，但过细账单、实时进度条和月度承诺会增加维护负担，也可能把兴趣项目变成隐性的履约关系。

证据：当前站内表达“让站点继续维持”，但没有说明支持用途或公开口径。

方向与要点：

- 默认只公开成本类别和大致月度区间，不公开供应商账号、账单明细或精确基础设施信息。
- 不做实时募款进度条，不把某月未达目标写成停站威胁。
- 如果不愿长期更新数字，可以只写“用于补贴维护成本”，并将本事项标记为 `cancelled`；主支持路径不受影响。
- 若未来成本结构明显变化，再手工更新，不建立自动财务同步。

完成判断：用户已经选择公开口径；若公开，内容准确、克制且不泄露敏感信息；若不公开，取消理由已记录。

Review 关注：检查数字是否容易过时，是否无意中披露基础设施规模或制造道德压力。

执行证据：尚无。

### SUP-006 — 建立低负担复盘与回馈边界

- 状态：`todo`
- 优先级：`P2`
- 波次：Wave 2
- Drive AI：`unassigned`
- Review AI：`unassigned`
- 依赖：SUP-003、SUP-004
- 最后更新：2026-07-16
- 结论置信度：`recommended`

问题与影响：没有复盘时，很难知道渠道是否仍值得保留；复盘和回馈过重时，又会把个人项目变成持续运营任务。

证据：当前没有维护成本支持的记录规则或回馈边界。

方向与要点：

- 每季度或每半年手工看一次渠道是否可用、支持金额大致能覆盖多少成本、是否产生额外事务；不新增用户行为追踪。
- 只记录汇总金额和渠道成本，不在仓库保存支持者名单、订单或联系方式。
- 可选回馈限于致谢、公开更新记录或路线说明，并允许支持者选择匿名。
- 不提供核心功能抢先权、付费专属功能、定制开发、优先排障或保证响应时间。
- 连续一段时间无人使用某渠道时，可以关闭该渠道，但无需因此恢复广告。

完成判断：复盘频率和记录范围已确定；支持渠道即使收入有限，也不会制造超过其价值的维护工作。

Review 关注：确认复盘没有演变成新的分析系统，回馈没有形成难以退出的长期承诺。

执行证据：尚无。

## 6. 明确不做

- 不新增或优化 AdSense 广告位，不开启移动端 anchor、vignette、in-feed 或全站 Auto Ads。
- 不为了证明撤广告合理而继续采集 coverage、RPM 或收入数据。
- 不设置付费墙、下载限额、上传额度或支持者专属核心功能。
- 不使用弹窗、悬浮催款条、倒计时、目标缺口警告或高频提醒。
- 不售卖会影响公共路线优先级的“功能投票权”。
- 不承诺以固定更新频率、长期在线时限或人工响应 SLA 交换支持。
- 不在仓库记录支持者个人信息或收款明细。
